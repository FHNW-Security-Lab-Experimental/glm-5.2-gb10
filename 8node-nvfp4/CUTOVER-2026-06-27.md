# GLM-5.2-NVFP4 + MTP on all 8 DGX Sparks — cutover & runbook (2026-06-27)

Replaced the two-model split (MiMo-V2.5 on 101-104 + MiniMax-M3 SGLang on 105-108)
with the **full `nvidia/GLM-5.2-NVFP4` on all 8 GB10 nodes at TP=8**, with in-checkpoint
MTP speculative decode and native sm_121 DSA sparse attention. This is the authoritative
account of what's live, how it was built, every fix and *why*, and what's still open.

## How it was built

1. **Weights** — `nvidia/GLM-5.2-NVFP4` (465 GiB) + `nvidia/MiniMax-M3-NVFP4` (250 GiB)
   downloaded on the head, then fanned to all 8 nodes over the 200G RDMA fabric
   (`scripts/distribute_models_rdma.sh`). HuggingFace throttles per-IP to ~42 MB/s; the
   reliable path was `hf_transfer` with Xet **disabled** (`hf_xet` ~1 MB/s; aria2 403s on
   Xet's 596 MiB byte-range-locked signed URLs). ~25 min RDMA fan-out for ~4.6 TB.
2. **Image** — `vllm-node-tf5-glm52:base` built on the head via eugr's `build-and-copy.sh
   --vllm-ref ab66606 -t ... --tf5` (prebuilt flashinfer/vLLM wheels), then `docker save |
   load` to all 8 over RDMA.
3. **Per-node staging** — NCCL 2.30.4 aarch64 → `~/models/nccl-2.30.4/libnccl.so.2`;
   CosmicRaisins Triton kernels → `~/glm-triton/`; `glm52-sparse-patches.sh` →
   `~/vllm-glm52/runtime/` (byte-identical sha256 across all 8); b12x 0.23.0 wheel →
   `~/models/wheels/`.

Ported from `CosmicRaisins/glm-5.2-gb10` (proven on 4 nodes / TP=4 / AWQ-15%-pruned + a
separate INT4 MTP draft). Our adaptation: **full NVFP4 on 8 nodes (no prune)** + the
**in-checkpoint** MTP draft the nvidia export ships (no separate draft model needed).

## The runtime stack (what differs from stock vLLM)

- **Kernels runtime-mounted** RO over the image's `vllm/v1/attention/backends/mla/` +
  `ops/deepseek_v4_ops/` (from `~/glm-triton/`).
- **`glm52-sparse-patches.sh`** runs in each container before `vllm serve` (idempotent,
  anchor-guarded vs `ab66606`, byte-identical on all 8) — three GB10/sm_121 fixes:
  - **STEP A (`glm52-sm12x-sparse`)** — wires the sm12x Triton sparse-MLA kernels;
    patches `vllm/utils/deep_gemm.py` to short-circuit `fp8_fp4_mqa_logits` /
    `fp8_fp4_paged_mqa_logits` / `tf32_hc_prenorm_gemm` to the sm12x fallbacks *before*
    the DeepGEMM `_missing()` gate; relaxes `SparseAttnIndexer` so sm_121 doesn't require
    `has_deep_gemm()`. STEP B installs b12x 0.23.0 + probes it to pick cudagraph mode.
  - **STEP A4 (`GLM52_SM12X_DECODE_REROUTE`)** — the 512k fix (see below).
  - **STEP C (`GLM52_NVFP4_MOE_BACKEND`)** — the MoE crash root-fix (see below).
- **NCCL 2.30.4** `LD_PRELOAD`ed (2.29.7 had an aarch64 `shm_broadcast` wedge) +
  `--device /dev/infiniband --cap-add IPC_LOCK --ulimit memlock=-1:-1` so NCCL uses
  RoCE/IB (`NET/IB`, not TCP). **This + NCCL 2.30.4 is what makes TP=8 stable** where our
  prior bare TP=8 GLOO-wedged.
- **In-checkpoint MTP** — `--speculative-config '{"method":"mtp",
  "num_speculative_tokens":3,"attention_backend":"FLASHMLA_SPARSE"}'` (no separate draft).
- **Per-node RDMA env** — head `enp1s0f0np0`/`rocep1s0f0`, workers
  `enP2p1s0f0np0`/`roceP2p1s0f0`; `GLOO/TP/UCX_SOCKET_IFNAME` = each node's own rail iface,
  never collapsed. `--kv-cache-dtype fp8_ds_mla`, `--hf-overrides '{"qk_rope_head_dim":64}'`
  (the head_dim=192/qk_rope=64 alias collision), `--enforce-eager`, parsers `glm45`/`glm47`.

## The fixes, and why (the journey)

1. **Port conflict** — GLM binds `:8000` with `--network host`, but `model-router` already
   held `:8000` → rank0 died `OSError Errno 98` *and* the `/health` poll got a false 200
   from the router. Fix: `model-router` added to the launcher's `STOP_CONTAINERS`.
2. **512k crash → `top_k_per_row_decode` reroute (STEP A4).** The DSA *decode* indexer
   called `torch.ops._C.persistent_topk`; on GB10 it needs `total_ctas =
   ceil(max_model_len/8472)` cooperative CTAs (31@256k, **62@512k, 124@1M**) >
   `num_sms*occupancy=48`, and its `FilteredTopK` fallback needs ≥128 KB smem vs sm_121's
   99 KB → `RuntimeError`, killing the engine. **Gated by configured `max_model_len`, not
   prompt length** (so 256k was always fine). Fix: on cap-family 120, route decode to the
   in-tree `else` branch `ops.top_k_per_row_decode` (set-equivalent top-k, 8 KB smem, one
   block/row). **Validated**: decode works at 512k, 0 `persistent_topk` errors, a
   **300,040-token needle retrieved correctly**, ~22.5 tok/s (no regression).
3. **NVFP4 MoE illegal-memory-access → MARLIN (STEP C).** The `FLASHINFER_CUTLASS` NVFP4
   MoE (TRT-LLM `cutlass_fused_moe` → `cudaMemsetAsync(final_output)`) intermittently
   CUDA-IMA'd and killed EngineCore (~1/4 h, and at boot warmup under cold-cache pressure).
   The global `--moe-backend marlin` flag is rejected by GLM's **unquantized** in-checkpoint
   MTP MoE, so the fix is **NVFP4-oracle-only**: on cap-family 120, strip FlashInfer **and**
   `VLLM_CUTLASS` from `fused_moe/oracle/nvfp4.py::AVAILABLE_BACKENDS` so it lands on
   **MARLIN**. MARLIN is **w4a16** (dequant-in-GEMM, bf16 activations) — a different
   codepath that never touches the broken FP4 grouped-GEMM, and correct (discards
   activation scales by design → the NVFP4 global-scale-factor concern is moot).
   `VLLM_CUTLASS` is **not** a valid alternative — it shares the broken SM120 kernel →
   ~5 tok/s garbage. **Validated**: MARLIN selected (`['MARLIN','EMULATION']`), coherent +
   correct output (reasoning/code/needle, NOT garbage), ~22.5 tok/s (no regression),
   survived a 6-way concurrent stress burst (67 reqs, 0 errors, 0 IMA). MARLIN's slightly
   larger w4a16 weights need `GPU_MEMORY_UTILIZATION=0.78` for 512k (at 0.72 the KV pool
   was ~0.7 GiB short — clean `ValueError`, no wedge; 0.78 → 32.89 GiB KV).

## Safety net & recovery

- **`sparks-glm52-watchdog`** (`tools/watchdog_glm52_cluster.sh` +
  `tools/systemd/sparks-glm52-watchdog.{service,timer}`, deployed to the head, enabled):
  every 60s, restarts GLM via `launch.sh` **only** when `/health` is dead AND the
  in-container `vllm serve` process is gone (true engine death — never during a slow
  prefill, which keeps the process alive). 2-strike, shares `restart.lock` (flock). A
  watchdog restart goes through `launch.sh` so it re-applies all patches incl. MARLIN. The
  stale `sparks-vllm-watchdog` / `sparks-kimi-watchdog` timers are **disabled** so they
  can't fight GLM.
- **Hard node recovery** (ping-only / sshd dead): Shelly power strips
  `http://10.84.93.20` + `http://10.84.93.21` (digest `admin:<key>`, 4 ports each = 8
  Sparks), `Switch.Set?id=N&on=false|true`. Needs the dgx-sparks-wg + fhnw VPNs.

## Context: 512k vs 1M

- **512k is the production ceiling.** Validated; ~22.5 tok/s; KV pool ~601k tokens.
- **True 1M is a SOFTWARE gap, not hardware.** The cluster has the memory (8×121=968 GiB;
  1M needs ~555 GiB with *sharded* KV), but vLLM **replicates** the MLA latent KV on every
  rank (~50 GiB/node @1M — MLA has 1 KV head so TP can't shard it), which overflows a
  121 GiB node. The KV-sharding lever, **decode-context-parallel (DCP)**, is a verified
  dead end for DSA: it hard-refuses fp8 KV (`mla_attention.py:788`), the DSA indexer has no
  cross-rank top-k (sharding would silently corrupt attention), and the sparse path returns
  no LSE. 1M would need a **sparse-aware context-parallel KV** implementation (shard fp8 KV
  + distributed global top-2048 + cross-rank KV gather + fp8 LSE combine) — upstream/
  research-grade work (cf. vLLM RFC #30055), not a config change. Details + the design
  questions are in `glm52-nvfp4-decode-research-2026-06-27.md`.

# Launch / production (from the head):
CONFIRM_GLM52=YES ENFORCE_EAGER=1 MAX_MODEL_LEN=524288 MAX_NUM_SEQS=4 \
  GPU_MEMORY_UTILIZATION=0.78 ~/vllm-glm52/runtime/launch.sh

~/vllm-glm52/runtime/launch.sh --stop        # reap the container on all 8
