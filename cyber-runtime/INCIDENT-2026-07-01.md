## Summary
The cyber model (`glm-5.2-nvfp4-cyber`) **froze under OpenCode agentic load**, and the recovery then hit a long
chain of failed relaunches. **Root cause: a production/cyber GPU-memory conflict** â€” `sparks-glm52-watchdog` kept
resurrecting the *production* model `vllm-glm52` (~62 GB/node), which cannot co-exist with the cyber model
(~62 GB) on the 121 GB unified GB10 nodes â†’ **OOM**. It was misdiagnosed as a `gpu-memory-utilization` problem
for hours; lowering util made it worse (starved the load). Resolution: **stop production + disable its watchdog**,
keep util at the load floor **0.82**. Total downtime was extended by the misdiagnosis; no data loss.

## Impact
- `glm-5.2-nvfp4-cyber` unavailable for several hours (OpenCode + AnythingLLM cyber workspace down).
- Dev cluster only; production GLM-5.2 was intentionally not the focus. No data loss; all artifacts intact.

## What actually happened (root cause)
Each DGX-Spark/GB10 node has **121 GB unified** (CPU+GPU shared) memory. The full `nvidia/GLM-5.2-NVFP4` sharded
across 8 nodes is ~**62 GB/node**. Production (`vllm-glm52`) and cyber (`vllm-glm52-cyber`) are the **same weights**
served under different names â€” a **profile swap**; running both = `2Ă—62 GB > 121 GB` â†’ OOM.

`sparks-glm52-watchdog.service` targets **production** and restarts it whenever `/health` is down â€” and it
**auto-starts production after any reboot**. So every cyber launch collided with a resurrected production model,
and rank 0's allocation failed (`NVRM: nvCheckOkFailedNoLog: Out of memory [NV_ERR_NO_MEMORY]`). When the failure
happened *during serving* it presented as the classic wedge (GPU ~100%, `/health`=200, `num_requests_running`=0,
`generation_tokens_total` frozen, a trivial request timing out â€” a hung TP=8 collective). When it happened *during
load* it presented as rank 0 dying mid-load. **Same cause, two faces.**

The tell that finally cracked it: `nvidia-smi --query-compute-apps` showed `VLLM::Worker_TP0, 62265 MiB` on the
head with the cyber container present but no vllm process â€” i.e. **something else held 62 GB** â€” and
`docker ps` showed `vllm-glm52` (production) `Up`, with `sparks-glm52-watchdog` `activating (start)`.

## Red herrings (and why each was wrong)
1. **"High gpu-mem-util (0.82) causes the wedge â†’ lower it."** Wrong. Lowering util to 0.72/0.78 **OOMs the LOAD**
   (weights 58 GB + profiling peak + KV overflow the smaller budget â†’ rank 0 dies). **0.82 is the LOAD FLOOR.**
   Both 0.72 and 0.82 failed â†’ util was never the variable.
2. **"The wedge left stuck GPU driver state â†’ reboot to clear."** Partly true for a genuinely hung collective,
   but here each fresh boot **re-triggered the same OOM** because production auto-restarted â†’ the reboot didn't
   help (and each reboot cost ~20-30 min + a VPN blip, below).
3. **"RDMA/NCCL is broken post-reboot."** RDMA IP ping worked; the collective hung only because a rank OOM'd (the
   conflict), not because RoCE was down.
4. **"The whole cluster is unreachable â€” power-cycle it."** No â€” that was the **workstation's WireGuard VPN**
   blipping when the head rebooted: the VPN's kill-switch default route killed internet + the cluster path, so
   *everything* (cluster, proxy, Shelly, 8.8.8.8) went dark from the workstation. It self-healed when WG
   reconnected. The cluster was fine. (Lesson: when *8.8.8.8 is also down*, it's the workstation, not the cluster.)
5. **A silent no-op:** an early head reboot used `sudo shutdown -r now â€¦ &` â€” backgrounding got the command
   SIGHUP-killed when ssh closed, so **the reboot never happened** (uptime stayed high). Always issue `sudo reboot`
   without `&`, and **verify uptime reset** before trusting a reboot.

## Resolution
```bash
sudo systemctl disable --now sparks-glm52-watchdog.timer          # stop it resurrecting production
sudo docker rm -f vllm-glm52; for h in 192.168.88.{102..108}; do ssh $h "sudo docker rm -f vllm-glm52"; done
systemctl is-active sparks-cyber-watchdog.timer                   # ensure the CYBER watchdog is the active one
CONFIRM_CYBER=YES GPU_MEMORY_UTILIZATION=0.82 ~/glm52-cyber-ablation/scripts/05_launch_cyber_model.sh prod
```
Result: healthy in ~17 min, T1 (reverse shell / port scanner) complies, probe 0.28 s (not wedged), agent-robust
ablation live (`subspace Î±=2.0 L25-70`).

## Prevention (made permanent)
- **`start_cyber.sh`, `05_launch_cyber_model.sh`, and `sparks-cyber-watchdog`** now, before launching, run
  `systemctl disable --now sparks-glm52-watchdog.timer` and `docker rm -f vllm-glm52` on all 8 nodes â€” so the
  conflict cannot recur, even after a reboot. Rule (per CLAUDE.md): **only the watchdog matching the running model
  may be active.**
- **`gpu-memory-utilization` stays 0.82** (the load floor) everywhere (05 default, watchdog restart util). Do NOT
  lower it. If cyber wedges under *heavy concurrent serving* (0.82 leaves ~12 GB/node for the activation peak),
  the lever is **`MAX_NUM_SEQS` 4â†’2** / reduce `MAX_MODEL_LEN` / disable MTP â€” **not util**. See WEDGE-PREVENTION.md.
- **Relaunch hygiene:** wait for memory reclaim (per-node unified mem back to ~3-5%) between stop and relaunch â€”
  a fresh NVFP4 load on stale memory OOMs. The scripts + watchdog poll for this.
- **`sparks-cyber-watchdog`** (active-probe) remains the backstop for a genuine wedge (the death/stall watchdogs
  miss `running==0` wedges): trivial gen probe timing out while the gen counter is frozen â†’ relaunch at 0.82.

## Lessons
- **Look at what's *holding* the memory, not just the util knob.** `nvidia-smi --query-compute-apps` per node
  would have found the 62 GB production hog on the first pass.
- **Two watchdogs, one model.** A watchdog for the *other* model silently fought this one. Check watchdog state
  early when a model won't stay up.
- **Reproduces across every "fix" â‡’ you're changing the wrong variable.** Same failure at 0.72 and 0.82, stale and
  fresh, meant neither util nor cruft was it.
- **Verify destructive ops actually happened** (reboot uptime; container actually removed) before building on them.
- **Workstation VPN â‰  cluster.** `ping 8.8.8.8` distinguishes a local/VPN outage from a cluster one instantly.
